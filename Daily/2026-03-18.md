### Call com Lemes — Planejamento de msgs diário (08:30–09:30)

Participantes: Fernando, Caíque, Giovane, Lemes, Lucas

Discussão sobre reestruturação do sistema de envio de campanhas na API oficial. Os principais pontos foram:

- **Performance do banco**: os critérios de seleção de público (ex: LTV mais alto) foram removidos porque geravam picos de uso no banco a cada consulta, inviável com milhares de empresas ativas. Hoje envia para quem chega primeiro.
- **Peso de prioridade**: na API oficial, foi introduzido um sistema de peso (baixa, média, alta) nas campanhas para gerenciar a distribuição diária de envios.
- **Fim da limitação de banimento**: como a Meta agora determina o volume de envios, não há mais o limitante antigo. Abre oportunidade para usar critérios mais inteligentes (dia preferido de compra, potencial de compra).
- **Proposta MCP + Cloud AI**: usar conector MCP com Cloud AI Platform para priorizar envios com base em critérios como última mensagem recebida, valor gasto e dia preferido de compra. Contorna a limitação do NECT ser atualizado apenas 1x/dia.
- **Travar segmento após publicação**: necessidade de impedir edição do segmento após a campanha ser publicada — troca de segmentos causa inconsistência nos funis e indicadores.
- **Campanhas baseadas em objetivos com IA**: ao invés de o usuário escolher segmentos, a criação seria guiada pelo objetivo (ex: aumentar ticket médio, esvaziar estoque) e a IA definiria o público automaticamente.
- **Tags em clientes**: Caíque sugeriu usar tags para marcar quem participou de campanhas — boa forma de segmentar.
- **Limite semanal vs diário**: limite de 30 msgs/dia é remanescente da API não oficial e precisa migrar para limite semanal.
- **Bug Italin House Dourados**: cron job enviou 149 mensagens com limite de 78. Hipótese: fila não esvaziou antes do próximo cron rodar, gerando envio duplicado. Lemes vai investigar com a IA.

## Reflexões / Aprendizados

O dia foi dominado por integrações e seus limites técnicos. Três conversas diferentes (Cloudfy, PDV-Legal, Italin House) tocaram no mesmo tema: como lidar com as restrições de APIs de terceiros sem comprometer a experiência do cliente. A lição que fica é que precisamos de um framework mais claro para decidir quando investir em soluções temporárias vs. esperar uma solução robusta — hoje essa decisão está sendo tomada caso a caso, sem critério consistente.

A call com Lemes trouxe ideias promissoras sobre IA para priorização de envios. O conceito de campanhas baseadas em objetivos com seleção automática de público tem potencial para ser um diferencial competitivo forte. Mas é uma mudança grande de paradigma — hoje o usuário escolhe manualmente quem recebe, amanhã a IA decidiria. Preciso pensar em como fazer essa transição gradual sem assustar os clientes atuais.

O bug de envio duplicado da Italin House é um alerta: à medida que crescemos na API oficial, a robustez dos jobs de envio precisa acompanhar. A hipótese do cron job rodando antes da fila esvaziar é plausível e relativamente simples de resolver com uma trava (tipo um campaign_key como sugeriu o Lemes), mas o fato de ter passado despercebido mostra que precisamos de melhor monitoramento.

> [!tip] Insight do dia
> A decisão "fazer rápido agora vs. fazer direito depois" apareceu em pelo menos 3 contextos hoje. Vale criar um framework leve (tipo uma decision matrix com critérios como impacto no cliente, dívida técnica gerada, tempo de vida da solução) para padronizar essas escolhas no time.
